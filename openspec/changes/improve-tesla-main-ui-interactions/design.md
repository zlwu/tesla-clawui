## Context

当前前端已经采用稳定 DOM 壳体、局部 patch、SSE streaming 和 Tesla 键盘保守避让。主界面的主要短板不在协议或渲染架构，而在交互状态的表达方式：消息列表只有隐式自动跟随，没有显式“回到底部”入口；textarea 在流式回复期间会整体禁用；头部状态 pill 占用垂直空间，却离当前操作最远；当前键盘避让主要依赖固定比例 fallback 与 `scrollIntoView()`，在 Tesla 车机以及 iOS Safari/Chrome 上仍可能出现输入框与系统输入法面板互相遮挡；`focusout` 后固定强制滚到底部，也会打断用户查看历史的上下文。

这次变更的目标不是模仿 ChatGPT iPad 的完整产品形态，而是借用其已验证的聊天节奏和移动网页布局原则，并筛掉 Tesla 浏览器不适合的部分。保留 Tesla 文本优先、系统语音输入优先、SSE 主链路和稳定 DOM 前提，只优化主界面“读消息、写消息、回到底部、等待回复”和“键盘弹出不遮挡操作”的节奏感。

## Goals / Non-Goals

**Goals:**
- 给消息列表增加显式的“回到底部/恢复跟随”控制，避免用户离开底部后失去方向。
- 把关键状态反馈收拢到 composer 附近，让发送、流式中、错误恢复都围绕当前操作发生。
- 在不支持并发请求的前提下，允许用户在助手流式回复期间继续编辑下一条草稿。
- 用分层 viewport 策略系统性解决 Tesla 与 iOS Safari/Chrome 上的输入框遮挡问题，保证 textarea、发送动作和关键状态在键盘弹出时仍可见。
- 改进 Tesla 键盘收放后的滚动保持逻辑，减少被动跳到底部造成的上下文丢失。
- 继续保持稳定 DOM 壳体和低复杂度实现，避免 Tesla 浏览器因为重排或节点替换而丢焦点。

**Non-Goals:**
- 不引入网页录音、ASR 上传、TTS 控制、手机端协同或多栏会话导航。
- 不改后端 API、消息持久化协议或 SSE 事件格式。
- 不做 React / 虚拟 DOM / WebSocket 重构。
- 不追求完全复刻 ChatGPT iPad 的视觉样式，只吸收适合 Tesla 的交互原则。

## Decisions

### Interaction Contract

主界面交互需求收敛为以下稳定契约，后续实现必须整体满足，而不是零散优化单点样式：

1. 消息区、composer、状态区始终处于单列主流程布局中，不能切换成覆盖式浮层模型。
2. 用户一旦离开底部，系统必须停止强制跟随，并提供明确的恢复入口。
3. 输入、发送、等待回复、错误恢复必须围绕 composer 就近反馈，减少抬头寻找状态。
4. 用户发送后直到首个流式增量到达前，必须有明确的“等待智能体响应”状态，避免界面看起来像卡住。
5. 键盘弹出时的首要目标不是“整个页面都可见”，而是“textarea、发送动作、关键状态仍可操作”。
6. 键盘收起时的首要目标不是“总是回到底部”，而是“保持用户当前上下文，只在明确需要时恢复跟随”。

这些契约参考了 ChatGPT iPad / 移动网页的有效交互原则，但做了 Tesla 化约束：不引入多栏、覆盖式底部工具条、复杂手势或重型动画。

### 1. 用显式消息跟随模式替代纯隐式自动滚动

前端将把当前消息区状态明确区分为“跟随最新内容”和“脱离底部查看历史”两种模式，并在脱离底部时显示独立的“回到底部”按钮。该按钮既承担恢复自动跟随的入口，也承担流式回复期间的方向提示。

这样设计的原因是当前实现仅靠 `distanceToBottom <= 48` 判断是否继续自动滚动，用户一旦离开底部就缺少显式恢复入口，只能手动滑回。ChatGPT iPad 的有效做法不是永远强制跟随，而是在阅读历史时尊重用户位置，并给出清晰的返回路径。

备选方案：
- 继续使用纯阈值自动恢复。否决，因为它对 Tesla 大字布局不够可感知。
- 每次有新消息都强制拉回底部。否决，因为会破坏历史阅读。

### 2. 保留极简头部，把运行状态迁移到 composer 附近

头部继续保留产品名和必要的轻量状态，但把“发送中 / 已离线 / 可恢复错误 / 已离开底部”等强交互反馈移到 composer 下方或上方的就近状态区。错误恢复按钮也与该状态区合并，避免用户在底部输入、却要抬头理解状态。

这样做可以减少纵向空间占用，并让状态与用户当前动作形成更短反馈回路。Tesla 竖向可用空间有限，头部每增加一行状态，就会直接压缩消息区和输入区。

备选方案：
- 保持现有头部 pill 样式，仅做文案调整。否决，因为主要问题是布局位置，不是文案。
- 完全去掉头部状态。否决，因为网络/会话状态仍需要最小可见性。

### 3. 流式回复期间允许继续编辑草稿，但保持单请求发送门禁

`isSendingText` 不再禁用 textarea，而是只禁用发送动作，并在 composer 状态区提示“正在回复，可先整理下一条”。这样用户可以在等待回复时继续输入或通过 Tesla 系统语音输入补全下一条内容，但不会触发第二个并发请求。

这样设计兼顾了 ChatGPT iPad 的“等待时仍可准备下一条”体验和当前后端的单请求假设。它不需要协议变更，只需要把“发送中”从输入禁用变为动作禁用。

备选方案：
- 维持 textarea 全禁用。否决，因为等待时段完全空转，车机输入成本高。
- 支持真正的并发多请求。否决，因为超出当前 MVP 与后端语义。

### 3.1 发送后到首个流式增量前需要独立等待态

发送消息成功发起后，到服务端返回首个流式 `delta` 之前，UI 需要显示独立的等待状态，例如“正在等待回复”或等价的轻量 loading 提示；一旦收到首个增量，再切换成“正在回复”的 streaming 状态。这样用户可以区分“请求已发出、模型尚未开始输出”和“模型已经开始持续返回内容”。

这样设计的原因是当前只有一个笼统的 thinking/sending 感知，用户很难判断请求是否已经发出、是否只是首字节较慢，还是页面真的失去响应。把等待首包和 streaming 输出分开，是成熟聊天产品常见的反馈细节。

等待首包的视觉表现采用轻量 ASCII 点动画，例如 `.` / `..` / `...` 的低频轮替，而不是复杂 spinner 或强视觉骨架屏。这样做更适合 Tesla 大字低噪音界面，也更稳妥地兼容 iOS Safari / Chrome 和 Tesla 浏览器的低复杂度渲染环境。首个 `delta` 到达后，该动画必须立即停止，并让位给真实流式文本；若请求失败，则直接切换为错误提示与恢复动作，而不是先闪回空白占位。

### 4. 键盘收放后的滚动恢复改为条件保持，而非 blur 后强制到底

当前 `focusout` 后固定调用 `scrollMessagesToBottom({ force: true })`，会把用户从历史阅读位置拉回底部。新的策略是记录 focus 前是否接近底部，以及 keyboard fallback 是否因为输入而抬升布局；只有原本就在底部或明确触发“回到底部”时，才在 blur 后恢复跟随。否则维持当前阅读位置，只回收键盘占位。

这样可以兼顾 Tesla 上 keyboard fallback 的稳定性和阅读上下文保持，避免“刚看上面一条消息，收起键盘就被拽走”的问题。

备选方案：
- blur 后永远不动滚动位置。否决，因为在底部发送消息的主路径下，键盘收起后仍需要顺滑回到底部。
- 仅依赖 `visualViewport`。否决，因为 Tesla 真机上该能力并不总是可靠。

### 5. 键盘避让采用“布局层级 + viewport 信号分层”的策略

主界面将继续保持 ChatGPT 移动网页同类产品常见的整体布局思路：应用根节点占据当前可用 viewport，高层 shell 使用单一滚动消息容器，composer 保持在正常文档流底部而不是 `position: fixed` 覆盖层。键盘弹出时，前端按三层信号处理：

1. 优先使用浏览器真实 layout viewport 缩小后的高度，例如 `interactive-widget=resizes-content` 配合根容器高度同步。
2. 如果 layout viewport 没有可靠缩小，则使用 `window.visualViewport.height` / offset 估算可用空间。
3. 如果两者都不可靠，再退回 Tesla 现有 focus 驱动的保守底部 inset，并只把 composer 滚进可视区。

这样做的原因是“输入框遮挡”本质上是 viewport 和输入法面板之间的空间同步问题，单靠一次性 `scrollIntoView()` 或固定比例 padding 无法稳定覆盖 Tesla 与 iOS 浏览器差异。分层策略能让支持新行为的浏览器直接受益，同时保住 Tesla 老浏览器的兜底稳定性。

备选方案：
- 把 composer 改成固定吸底浮层。否决，因为系统键盘弹出时最容易与 safe area、消息区滚动和 Tesla 焦点状态互相冲突。
- 只靠 CSS `100vh` / `100dvh`。否决，因为不同浏览器对键盘弹出时的 viewport 语义并不一致，仍需要 JS 观测兜底。

### 6. 键盘避让按“优先级明确、补偿不叠加”的规则计算

键盘相关布局补偿必须遵守单一路径生效原则，避免同时吃到 layout viewport resize、`visualViewport` inset 和 focus fallback，造成双重抬升。具体规则如下：

1. 如果 layout viewport 已明显缩小，则该高度就是主真值，不再额外叠加保守键盘 inset。
2. 如果 layout viewport 未缩小，但 `visualViewport` 可用，则只根据 `visualViewport` 推导一次可用底部空间。
3. 只有当前两类真实信号都不足时，才启用 focus fallback。
4. safe area inset 始终作为环境底边的一部分参与计算，但不得被重复累加到 keyboard inset。

这样做是为了把“避让策略”从经验型样式 patch 变成确定性的布局计算，避免不同设备上输入框忽高忽低、发送按钮被多次抬升或 composer 下方留下无法回收的空白。

### 7. 键盘过渡期采用“延迟收敛而非持续抖动”的策略

系统输入法弹出和收起过程中，viewport 信号可能在短时间内连续波动。前端不应每次都立即触发新的滚动和高度重算，而是允许一个短暂 settle 窗口，在该窗口内收敛到最后一个稳定值，再决定是否执行 `scrollIntoView()` 或恢复跟随。

这样设计是因为当前 Tesla 与 iOS WebKit 在键盘动画期间都可能出现多次 resize / viewport 回调；如果每次都立刻滚动，最容易造成 composer 抖动、消息区闪跳和焦点丢失。

## Risks / Trade-offs

- [Risk] 流式期间可编辑草稿，可能让用户误以为可以立即发送第二条。 → Mitigation: 发送按钮保持禁用，并在 composer 状态区明确提示“请等待当前回复完成”。
- [Risk] 新增“回到底部”入口会增加一个悬浮控件，影响 Tesla 小屏纯净度。 → Mitigation: 仅在脱离底部时显示，尺寸和层级低于 composer，不遮挡最近消息。
- [Risk] 更复杂的滚动模式可能在老浏览器上产生边界 bug。 → Mitigation: 继续使用单一滚动容器和显式 DOM 节点，补充单元测试与手动验证场景。
- [Risk] iOS Safari、iOS Chrome 和 Tesla 车机对键盘弹出时的 viewport 行为不一致。 → Mitigation: 采用 layout viewport / `visualViewport` / focus fallback 的分层策略，并补充跨端手动验证。
- [Risk] 头部状态收口后，某些故障信息的全局可见性下降。 → Mitigation: 保留最小网络/会话指示，把可操作状态放在 composer 附近。

## Migration Plan

1. 先补 spec，明确消息跟随、composer 状态和键盘上下文保持的目标行为。
2. 在前端 state 和 render 层增加显式交互状态，不改后端接口。
3. 调整样式与 DOM 引用，保持 textarea、message list、composer 根节点稳定，并把 viewport 计算收拢到单一来源。
4. 建立手动验证矩阵，至少覆盖 Tesla 车机、iOS Safari、iOS Chrome 的 focus、输入、发送、收起键盘、旋转或窗口变化场景。
5. 通过单元测试覆盖滚动模式、发送门禁、viewport 高度计算和状态反馈，再执行 `npm run lint`、`npm run typecheck`、`npm test`、`npm run build`。
6. 若真机或本地验证发现回归，可先回退到旧的滚动模式分支实现，不影响后端数据与协议。

## Open Questions

- “回到底部”按钮是否需要同时显示未读提示，还是只做纯动作入口。
- 流式中 composer 状态文案是否需要区分“首字节等待中”和“正在持续输出”。
- iOS Safari 与 iOS Chrome 上是否需要单独记录 keyboard overlap 观测日志，以便收敛 fallback 阈值。
- 现有 `MAX_VISIBLE_MESSAGES = 8` 是否足以支撑新的历史浏览体验，还是应在后续独立 change 中再评估可见窗口策略。
