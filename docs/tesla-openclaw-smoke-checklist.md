# Tesla OpenClaw MVP 冒烟清单

## 用途

用于执行 `T4.6` 目标设备实测。

本清单只验证 MVP 主链路：

```text
页面打开 -> session 创建 -> 文本兜底 -> 语音输入 -> 多轮显示 -> 弱网恢复
```

不验证：

- TTS
- 手机端协同
- SSE / WebSocket
- 复杂 UI 效果

## 环境准备

1. 项目根存在 `.env`
2. 已执行 `npm install`
3. 已执行 `npm run lint`
4. 已执行 `npm run typecheck`
5. 已执行 `npm test`
6. 已执行 `npm run build`
7. 开发态可执行 `npm run dev`，由 Fastify 统一提供页面与 API
8. 若用生产态验证，可执行 `npm run start -w @tesla-openclaw/server`
9. 默认访问地址为 `http://127.0.0.1:3000`
10. 若需要让 Tesla 访问本机环境，只可使用临时测试方式，例如 `cloudflared` quick tunnel；不要把它记录为正式方案

## 当前已知前置结论

- Mac Chromium 下本清单的本地预检已通过
- 真实 provider 联调已通过：`Qwen ASR + OpenRouter LLM`
- 当前剩余未完成项主要是 Tesla 真机兼容性与稳定性验证

## 执行顺序

### 1. 页面基础加载

1. 在 Tesla 浏览器打开页面 3 次
2. 确认页面能进入，不白屏
3. 确认页面自动创建 session

通过标准：

- 3 次都能进入页面
- 页面能显示“会话已连接”或等价状态

### 2. 文本兜底链路

1. 发送 1 条短文本
2. 发送 1 条稍长文本
3. 确认都收到 assistant 回复

通过标准：

- 文本请求成功
- 消息按顺序显示
- 大字文本清晰可读

### 3. 麦克风与录音

1. 首次点击语音按钮，观察是否弹权限或直接进入录音
2. 连续录音 10 次，每次 3~8 秒
3. 记录成功次数、失败次数、失败现象

通过标准：

- 建议至少 8/10 成功
- 成功时能拿到 transcript 和 assistant 回复
- 失败时页面可恢复，不会卡死在中间态

### 4. 语音上传与多轮

1. 在语音成功的前提下，连续进行 5 轮混合交互
2. 至少包含 3 次语音和 2 次文本
3. 观察滚动、消息顺序、状态切换是否正常

通过标准：

- 至少 5 轮稳定交互
- 最近消息显示正常
- 没有明显卡死、错序或重复消息

### 5. 刷新恢复

1. 在已有最近消息时手动刷新页面
2. 确认最近消息恢复
3. 再继续发送 1 条文本或 1 次语音

通过标准：

- 最近消息仍可见
- 刷新后还能继续交互

### 6. 弱网恢复

1. 在页面正常工作时断开网络
2. 再发 1 次文本或语音
3. 观察是否进入错误态并出现“重试上一步”
4. 恢复网络后点击“重试上一步”

通过标准：

- 出错文案可理解
- 页面不死锁
- 恢复网络后可继续使用

### 7. 长时间停留

1. 页面保持打开 15~30 分钟
2. 中间穿插少量交互
3. 观察是否自动刷新、回收或明显变卡

通过标准：

- 无明显崩溃
- 无频繁自动重载
