# Tesla OpenClaw MVP 冒烟清单

> 状态：测试执行参考文档。
> 与 OpenSpec 的关系：可作为手工验证清单，但不替代 change 内的任务与验收定义。

## 用途

用于执行 `T4.6` 目标设备实测。

本清单只验证 MVP 主链路：

```text
页面打开 -> session 创建 -> 系统语音输入法 / 文本输入 -> 发送 -> 多轮显示 -> 弱网恢复
```

不验证：

- TTS
- 手机端协同
- WebSocket
- 复杂 UI 效果

## 环境准备

1. 项目根存在 `.env`
2. 已执行 `npm install`
3. 已执行 `npm run lint`
4. 已执行 `npm run typecheck`
5. 已执行 `npm test`
6. 已执行 `npm run build`
7. 开发态可执行 `npm run dev`，由 Fastify 统一提供页面与 API
8. 若用生产态验证，可执行 `npm run start -w @tesla-openclaw/server`
9. 默认访问地址为 `http://127.0.0.1:3000`
10. 若需要让 Tesla 访问本机环境，只可使用临时测试方式，例如 `cloudflared` quick tunnel；不要把它记录为正式方案

## 当前真机验证结果

截至当前验证，Tesla 真机结论为：

`Proceed with Caveats`

已确认：

- 网页麦克风录音不可用
- 系统语音输入法 / 长按系统语音键输入可用
- 文本框 -> 发送 -> 大字文本多轮显示主链路可用

当前已知瑕疵：

- 输入框会被输入面板部分遮挡

## 执行顺序

### 1. 页面基础加载

1. 在 Tesla 浏览器打开页面 3 次
2. 确认页面能进入，不白屏
3. 确认页面自动创建 session

通过标准：

- 3 次都能进入页面
- 页面能显示“会话已连接”或等价状态

### 2. 文本兜底链路

1. 发送 1 条短文本
2. 发送 1 条稍长文本
3. 确认都收到 assistant 回复

通过标准：

- 文本请求成功
- 消息按顺序显示
- 大字文本清晰可读

### 3. 系统语音输入与文本输入

1. 聚焦输入框后尝试唤起系统键盘语音输入法
2. 连续使用系统语音输入 5 到 10 次，观察识别结果是否能稳定进入文本框
3. 手动编辑文本后发送，记录成功次数、失败次数、失败现象

当前结果：

- Tesla 真机网页麦克风权限不可用
- 系统语音输入法可作为可接受替代路径
- 本项不再作为真机首版 Go/No-Go 判定依据

### 4. 文本发送与多轮

1. 连续进行 5 轮混合交互
2. 可包含系统语音输入后的文本发送和手动文本输入
3. 观察滚动、消息顺序、状态切换和流式输出是否正常

当前结果：

- 通过系统语音输入法进入文本框后，主链路可继续工作
- 多轮显示功能可用

### 5. 刷新恢复

1. 在已有最近消息时手动刷新页面
2. 确认最近消息恢复
3. 再继续发送 1 条文本

通过标准：

- 最近消息仍可见
- 刷新后还能继续交互

### 6. 弱网恢复

1. 在页面正常工作时断开网络
2. 再发 1 次文本
3. 观察是否进入错误态并出现“重试上一步”
4. 恢复网络后点击“重试上一步”

通过标准：

- 出错文案可理解
- 页面不死锁
- 恢复网络后可继续使用

### 7. 长时间停留

1. 页面保持打开 15~30 分钟
2. 中间穿插少量交互
3. 观察是否自动刷新、回收或明显变卡

通过标准：

- 无明显崩溃
- 无频繁自动重载
