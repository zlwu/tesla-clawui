# Tesla 车机 OpenClaw 客户端开发任务拆解（TASKS）

## 1. 文档目标

本文档将 Tesla 车机 OpenClaw 客户端 MVP 的实现工作拆解为一组可执行任务，供人工开发或交由 Codex 实施。

当前任务拆解基于新的 MVP 定义：

> 在 Tesla 车机浏览器中实现“语音输入 + 大字文本多轮对话显示”。

本阶段：

- **不把 TTS 作为首版主链路前提**
- **不引入手机端协同**
- 只有在 Tesla 浏览器存在硬门槛时，才讨论 fallback 架构

目标：

- 任务边界清晰
- 依赖关系明确
- 每项任务都有可验证的验收标准
- 可按阶段推进，而不是一次性铺开

---

## 2. 项目范围（MVP）

MVP 只解决以下问题：

1. Tesla 浏览器可打开页面
2. 用户可发文本消息
3. 用户可录音并上传语音
4. 服务端完成 ASR -> LLM 主链路
5. 浏览器可展示大字文本回复并支持多轮继续
6. 错误时可恢复，弱网下不至于彻底卡死

MVP 暂不包含：

- 自动 TTS 播放
- 手机端协同
- 双端同步
- 持续监听
- 唤醒词
- WebRTC 双工语音
- 富文本聊天
- 复杂用户设置页
- 多账户体系
- 重型前端框架

---

## 3. 实施原则

1. 优先跑通最短主链路
2. 先稳定，再优化体验
3. 浏览器端保持极简
4. 服务端承担复杂度
5. 先定义类型与契约，再写实现
6. 所有模块都应便于 Codex 理解和修改
7. 只有硬门槛才触发 fallback 讨论

---

## 4. 开发阶段总览

- **Phase 0**：项目初始化与约束固化
- **Phase 1**：文本聊天主链路
- **Phase 2**：语音输入主链路
- **Phase 3**：多轮显示、状态机与稳定性
- **Phase 4**：日志、弱网恢复与交付准备
- **Phase 5（可选）**：TTS 与增强能力

- 创建根目录说明文件

  web/
  server/
  docs/
  package.json
  README.md

- 项目可被清晰识别为单仓结构

## 6. Phase 1：文本聊天主链路

## T1.1 定义共享类型与 API 契约

### 目标
确保前后端围绕统一契约开发。

### 任务
- 定义 session 类型
- 定义 message 类型
- 定义通用 error 类型
- 定义文本输入接口 schema
- 定义消息列表接口 schema

### 验收标准
- 关键请求/响应都有类型定义
- 前后端可复用同一份类型或严格镜像定义

---

## T1.2 实现会话创建接口

### 目标
前端加载时能够拿到一个可用 sessionId。

### 接口
- `POST /api/session/create`

### 任务
- 创建 session 记录
- 返回 sessionId / token / 初始状态
- 写入基础日志

### 验收标准
- 前端可成功创建 session
- 服务端数据库中有会话记录

---

## T1.3 实现文本输入接口

### 目标
用户可以通过文本与 OpenClaw 进行对话。

### 接口
- `POST /api/text/input`

### 任务
- 接收 sessionId + text
- 读取会话上下文
- 调用 LLM/OpenClaw provider
- 保存 user / assistant 消息
- 返回 assistant 文本

### 验收标准
- 能完成一次文本请求—响应
- 消息正确落库
- 出错时返回统一错误结构

---

## T1.4 实现消息列表接口

### 目标
前端可拉取最近消息用于展示。

### 接口
- `GET /api/messages?sessionId=...`

### 任务
- 支持按 sessionId 查询
- 只返回最近 N 条
- 返回纯文本消息内容

### 验收标准
- 前端刷新后能看到最近消息
- 返回条数受限
- 不返回无关字段

---

## T1.5 前端文本聊天 UI

### 目标
先跑通无语音的最小交互。

### 任务
- 实现大字消息列表区域
- 实现文本输入框
- 实现发送按钮
- 实现基础 loading / error 展示
- 页面默认创建 session

### 验收标准
- 页面打开可正常发送文本消息
- 回复能显示在消息区
- 字体清晰易读
- 刷新后可恢复最近消息

---

## 7. Phase 2：语音输入主链路

## T2.1 实现前端录音模块

### 目标
浏览器可完成一次录音并输出 Blob。

### 任务
- 封装 `MediaRecorder`
- 实现开始录音
- 实现停止录音
- 输出音频 Blob 与 mimeType
- 处理权限错误

### 验收标准
- 点击后可开始录音
- 停止后可获得有效 Blob
- 权限失败时有明确提示

---

## T2.2 实现语音上传接口

### 目标
服务端能够接收前端录音文件。

### 接口
- `POST /api/voice/input`

### 任务
- 支持 multipart/form-data
- 接收 sessionId、requestId、audio file
- 保存临时音频文件
- 记录 mimeType / 文件路径 / 上传时间

### 验收标准
- 能成功上传录音文件
- 服务端可读到音频文件
- 重复 requestId 可识别

---

## T2.3 实现 ASR provider adapter

### 目标
将上传音频转换为文本。

### 任务
- 定义 `AsrProvider` 接口
- 实现一个具体 provider
- 支持语言参数（默认中文）
- 返回识别文本与可选耗时

### 验收标准
- 给定音频文件可返回识别文本
- provider 错误能被统一捕获

---

## T2.4 打通语音输入完整链路（核心）

### 目标
实现：录音上传 -> ASR -> LLM -> 文本返回。

### 任务
- 将 `/api/voice/input` 串起 ASR 与 LLM
- 保存 user 文本（ASR 结果）与 assistant 回复
- 返回转写文本 + 回复文本

### 验收标准
- 用户录音后可获得文本回复
- ASR 文本能被保存为用户消息
- 服务端链路出现错误时能定位到具体阶段

---

## T2.5 前端语音按钮与基础状态展示

### 目标
用户可通过一个大按钮完成录音主流程。

### 任务
- 增加录音按钮
- 展示当前状态：idle / recording / uploading / transcribing / thinking / error
- 录音结束后自动上传

### 验收标准
- 不需要打开控制台也能看懂当前状态
- 录音操作简单明确
- 错误后能恢复到可继续使用状态

---

## 8. Phase 3：多轮显示、状态机与稳定性

## T3.1 实现前端状态机

### 目标
用显式状态控制语音主流程，避免逻辑散乱。

### 任务
- 定义状态枚举
- 实现状态切换函数
- 将录音、上传、识别、思考状态纳入统一管理
- 错误统一流入 `error`

### 验收标准
- 同一时刻只有一个主状态
- 状态流转可预测
- 错误恢复后能回到 `idle`

---

## T3.2 优化大字文本多轮显示

### 目标
让车机端在多轮对话时保持可读、不卡。

### 任务
- 限制最近消息条数（建议 6~8 条）
- 优化消息区布局与字号
- 保证长文本换行与滚动行为可控
- 区分 user / assistant 展示样式

### 验收标准
- 至少 5 轮连续对话仍清晰可读
- 页面不会因消息增加迅速卡顿

---

## T3.3 页面刷新与最近消息恢复

### 目标
页面刷新后仍能恢复最近上下文，避免体验断裂。

### 任务
- 刷新后重建 session 状态
- 拉取最近消息
- 恢复到可继续交互状态

### 验收标准
- 页面刷新后可继续当前会话
- 最近消息不会消失或错乱

---

## T3.4 页面性能与消息裁剪

### 目标
控制老款 Atom 车机上的渲染压力。

### 任务
- 限制一次渲染的消息数量
- 避免无意义重渲染
- 限制单条消息最大展示长度（必要时折叠）
- 确保纯文本渲染，不引入复杂 markdown 处理

### 验收标准
- 多轮对话后页面仍可用
- 无明显掉帧、卡死、白屏

---

## 9. Phase 4：日志、弱网恢复与交付准备

## T4.1 实现统一错误模型

### 目标
避免前后端对错误理解不一致。

### 任务
- 定义错误码枚举
- 定义统一错误响应结构
- 对 ASR / LLM / 上传失败做分类

### 验收标准
- 所有主要失败路径都有明确错误码
- 前端能按错误码决定提示文案

---

## T4.2 增加请求链路日志与耗时埋点

### 目标
可定位慢点与失败点。

### 任务
- 为每次请求生成 requestId
- 记录 sessionId
- 记录上传耗时
- 记录 ASR / LLM 耗时
- 记录总耗时

### 验收标准
- 任意一次失败请求都能通过日志回溯
- 能看出慢在哪一段

---

## T4.3 实现 requestId 幂等保护

### 目标
避免弱网或重复点击导致重复处理。

### 任务
- `/api/voice/input` 接收 requestId
- 对已处理请求返回已有结果或明确冲突状态

### 验收标准
- 相同 requestId 不会重复生成多份回复

---

## T4.4 弱网与断线恢复策略

### 目标
让页面在网络不佳时至少“可解释、可恢复”。

### 任务
- 前端识别网络失败
- 展示离线或失败提示
- 允许用户手动重试
- 失败后回到可继续交互状态

### 验收标准
- 网络失败不会让 UI 卡死在 loading
- 用户能手动恢复流程

---

## T4.5 编写 README / API 文档 / 测试清单

### 目标
让项目具备可交接、可持续开发能力。

### 任务
- 编写 README
- 编写 API 文档
- 编写测试计划与验收清单

### 验收标准
- 新开发者可按 README 启动项目
- 不看代码也能理解接口怎么调
- 有明确的冒烟测试清单

---

## T4.6 目标设备实测

### 目标
在 Tesla 浏览器上验证真实可用性。

### 任务
- 打开页面
- 创建 session
- 发文本
- 验证真机主输入路径（系统语音输入法 / 长按系统语音键输入）
- 记录网页内录音兼容性结论
- 验证 5~10 轮连续交互
- 记录兼容性问题

### 验收标准
- Tesla 浏览器上至少完成 5 轮稳定交互
- 记录已知限制与后续优化项

### 当前状态
- 已完成
- 当前结论：`Proceed with Caveats`
- 真机首版主路径已收口为系统语音输入法 / 长按系统语音键输入 -> 文本框 -> 发送
- 网页内录音在 Tesla 真机上降级为辅助兼容能力，不再作为首版 Go/No-Go 前提

## 11. 建议的任务执行顺序

建议按以下顺序推进：

1. T0.1 ~ T0.6
2. T1.1 ~ T1.5
3. T2.1 ~ T2.5
4. T3.1 ~ T3.4
5. T4.1 ~ T4.6
6. T5.1 ~ T5.2（可选）

### 原则
- 文本 streaming 已完成，后续排版增强应建立在现有 SSE 主链路之上
- 流式能力优先采用 SSE，不先引入 WebSocket
- 不要先做优化，再补主链路
- 不要跳过类型和契约直接堆实现
- 不要让 TTS 阻塞语音输入 + 文本显示主线

---

## 12. 建议 Codex 的工作方式

如果交给 Codex 执行，建议按以下方式推进：

1. **每次只做 1~3 个相关任务**
2. 每次提交前先跑 lint + test
3. 每完成一个阶段就产出简要变更说明
4. 不允许在未说明的情况下引入重型依赖
5. 遇到 Tesla 浏览器不确定兼容项时，先实现可回退方案

### 禁止 Codex 擅自做的事

- 引入 React / Next.js / Vue
- 大改项目目录结构
- 直接把前端做成 SPA 富应用
- 在浏览器端直连模型服务
- 跳过测试只求“看起来能跑”
- 擅自把手机端协同重新拉回 MVP

## 14. 一句话执行策略

> 先把语音输入和大字文本多轮跑通；只有这条主链路过不去，才值得讨论更复杂的 fallback。
