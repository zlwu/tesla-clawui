# Tesla OpenClaw 真机 UI 交互设计

## 1. 文档目标

本文档用于冻结 Tesla 真机输入区与聊天页面的交互设计，并同步当前已经落地的前端收口结果。

本设计的目标不是做一个独立的“新产品 UI”，而是基于真机验证结果，把当前 MVP 收口成：

- 更接近 ChatGPT 的聊天交互
- 更适合 Tesla 真机的输入方式
- 不偏离 Tesla 单端、语音优先、大字文本输出的产品边界

## 2. 设计输入

截至 2026-03-10，真机验证已经确认：

1. Tesla 浏览器内的网页麦克风权限不可用
2. 系统键盘的语音输入法可用
3. 长按系统语音键输入可作为同类主路径
3. 当前页面输入区在键盘/语音输入面板弹出时存在遮挡问题
4. 发送按钮与输入框距离过远时，真机上操作不顺手
5. 页面麦克风按钮与发送按钮过近时，容易形成误操作

因此，首版真机 UI 的主路径明确调整为：

```text
系统语音输入法 / 长按系统语音键输入 -> 文本框 -> 发送 -> 大字文本多轮显示
```

网页内录音仅保留为兼容能力，不再视为 Tesla 真机首版主路径。

## 3. 设计原则

### 3.1 总原则

- 交互逻辑参考 ChatGPT
- 页面结构必须继续保持轻量
- 输入区必须围绕 Tesla 真机操作习惯收口
- 输入与发送必须贴近，避免跨区域操作
- 错误提示不应抢占主输入区

### 3.2 不变边界

- 仍然是 Tesla 单端
- 语音依然是主入口，但 Tesla 真机主入口改为系统输入法语音输入或长按系统语音键输入
- 输出仍然是大字文本多轮显示
- 不引入 TTS、手机端协同、SSE、WebSocket、复杂设置页

## 4. 页面结构

页面保持单页结构，但收敛为更接近 ChatGPT 的三层：

1. 顶部轻量头部
2. 中部消息流
3. 底部固定 composer

### 4.1 顶部轻量头部

包含：

- 产品名
- 极简状态信息
- 在线/离线、会话状态弱化显示

不再保留大面积状态卡片，也不保留会在焦点切换时闪动的副标题。

### 4.2 中部消息流

目标：

- 接近 ChatGPT 的阅读体验
- 让 assistant 回复成为主视觉内容
- 用户消息与 assistant 回复有清晰区分

要求：

- 单列消息流
- assistant 回复优先阅读感
- 用户消息更轻、更紧凑
- 保持大字号与高可读性

### 4.3 底部固定 composer

这是本次改版的核心。

composer 必须始终包含：

- 输入框
- 条件显示的发送按钮
- 次级语音工具按钮
- 错误提示 / 恢复操作

composer 必须固定在页面底部，并在键盘弹出时整体上移。

## 5. 输入区交互设计

## 5.1 结构

推荐结构如下：

```text
[ 多行输入框                                  ][发送按钮]
  [次级麦克风按钮]  [错误提示 / 恢复操作]
```

其中：

- 输入框为主区域
- 发送按钮固定在输入框最右侧
- 麦克风按钮移出主输入行，放在 composer 下方次级工具位
- 没有文字时，发送按钮隐藏
- 发送按钮即使隐藏，也保留固定占位，不改变输入区布局宽度

## 5.2 语音按钮

语音按钮逻辑必须分平台理解：

### Tesla 真机主路径

- 用户长按系统键盘语音输入按钮，或使用系统语音键输入
- 语音识别结果进入输入框
- 用户点发送按钮发送

因此在 Tesla 真机上，页面麦克风按钮更像：

- 辅助入口
- 能力提示入口
- 或未来可恢复的网页内录音入口

### 网页录音可用时

如果浏览器支持网页内录音：

- 语音按钮可点击开始录音
- 录音中切换为停止状态
- 停止后提交到后端语音链路

### 网页录音不可用时

如果浏览器不支持网页内录音：

- 语音按钮保留，但不再作为主路径
- 不能抢占发送按钮的主操作位置

## 5.3 发送按钮

发送按钮采用 ChatGPT 风格逻辑：

### 显示条件

- `draftText.trim().length > 0`
- 当前不在发送中
- 当前不在录音中
- session 已就绪

### 隐藏条件

- 输入框为空
- 录音中
- thinking / uploading / transcribing 中

### 交互原则

- 一旦有文字，发送按钮成为主操作
- 没有文字时，页面不保留高存在感的发送按钮
- 发送按钮必须紧贴输入框，不能掉到底部另一行
- 发送按钮隐藏时不应造成布局跳动

## 5.4 输入框

输入框要求：

- 默认 2 行高度
- 最多扩展到 4 行以内
- 超过后内部滚动
- placeholder 需要直接承担短提示作用
- 当前收口文案为：
  - 支持网页语音时：`给 OpenClaw 发消息，也可使用系统语音输入后发送`
  - Tesla 真机场景：`给 OpenClaw 发消息，Tesla 真机可使用系统语音输入`

输入框是 Tesla 真机上的主输入容器，因此必须保证：

- 与发送按钮一起可见
- 聚焦后不被系统键盘大面积遮挡
- 长按系统语音输入后文本能直接确认与发送

## 6. 键盘激活态

当输入框聚焦时，页面进入 `keyboard-active` 模式。

### 目标

- 让输入框与发送按钮始终可见
- 让消息区仍保留最小阅读空间
- 减少无关元素占用

### 行为

1. composer 上移到键盘上沿
2. 顶部头部压缩
3. 非关键信息弱化
4. 消息区保留但缩小
5. 发送按钮继续贴在输入框右侧

### 真机优先规则

如果键盘弹出后空间过于紧张，应优先保证：

1. 输入框可见
2. 发送按钮可点
3. 最近 1~2 条消息仍可见

其他区块可以缩小甚至暂时弱化。

## 7. 错误与恢复

错误提示不再占据主输入区。

规则如下：

- 发送失败：在 composer 下方显示错误文案
- 可重试错误：显示 `重试上一步`
- 网页录音不可用：作为能力提示，不反复当作致命错误
- 网络错误：靠近输入区显示，但不把发送操作挤走

## 8. 视觉风格方向

本次视觉方向明确采用：

- ChatGPT 风格布局
- 极简头部
- 清晰单列消息流
- 圆角底部 composer
- 低干扰背景
- 以内容阅读和输入效率优先

同时保留 Tesla 场景要求：

- 字号偏大
- 对比度足够
- 按钮面积适合车机场景点击

## 9. 最终交互规则

以下规则作为重写实现时的冻结标准：

1. 页面只保留一个底部 composer
2. 发送按钮固定在输入框最右侧
3. 发送按钮只在有文字时显示，但隐藏时仍保留固定占位
4. 麦克风按钮降级为次级工具位，不与发送按钮并列抢主操作
5. Tesla 真机默认主语音入口为系统键盘语音输入
6. 长按系统语音键输入与系统键盘语音输入同属 Tesla 真机主路径
7. 网页内录音在 Tesla 真机上视为辅助能力
8. 键盘弹出时，输入框与发送按钮必须完整可见
9. 错误提示不能抢占主输入动作

## 10. 当前已落地实现

截至当前代码状态，以下 UI 收口已经完成：

1. 头部仅保留产品名与状态 pill
2. 副标题已移除，避免焦点切换闪动
3. 底部使用单一 composer
4. 发送按钮固定在输入框右侧，采用统一风格 SVG 图标
5. 麦克风按钮已移到 composer 次级工具位，并采用统一风格 SVG 图标
6. placeholder 已整合系统语音输入提示
7. 按钮事件命中已修正，支持点击 SVG 图标本体
8. 发送按钮显隐已改为固定占位，不再改变主布局宽度

## 11. 后续验证重点

接下来真机验证重点为：

1. 系统语音输入是否稳定写入文本框
2. 发送按钮在输入过程中是否始终可点
3. 麦克风按钮是否仍有误触风险
4. 键盘弹出时输入区是否仍有遮挡
