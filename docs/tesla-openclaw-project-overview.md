# Tesla OpenClaw 项目总览

> 状态：历史背景总览文档，不再作为后续 OpenSpec 开发主依据。
> 当前对应 OpenSpec：`tesla-browser-boundary`、`tesla-chat-ui`
> 说明：本文中仍出现的“网页录音 / ASR 主链路”表述已经过时；当前主链路固定为“系统语音输入法 / 长按系统语音键输入 -> 文本框 -> 发送 -> SSE 文本回复”。

## 1. 项目简介

这是一个面向 **Tesla 老款 Atom / MCU2 车机浏览器** 的 OpenClaw 客户端项目。

项目当前目标非常明确：

> 在 Tesla 浏览器中实现一个 **支持系统语音输入法辅助输入、支持大字文本多轮显示** 的极简对话客户端。

它不是一个通用 Web Chat 产品，也不是一个富前端 AI 应用，更不是一个双端协同系统。

当前阶段，项目只关心一件事：

> 让用户在车里，直接通过 Tesla 浏览器，和 OpenClaw 进行可用的多轮对话。

---

## 2. 当前 MVP 定义

MVP 的核心能力：

1. 页面可在 Tesla 浏览器中打开
2. 用户可通过文本输入与助手对话
3. 用户可结合系统语音输入法提高文本输入效率
4. 服务端完成文本输入 -> LLM 主链路
5. 车机端以 **大字文本** 显示最近多轮对话
6. 出错时可恢复，弱网下不至于彻底卡死

### 当前不纳入 MVP 的能力

- 自动 TTS 播放
- 手机端协同
- 双端同步
- Siri / 快捷指令集成
- 持续监听
- 唤醒词
- WebRTC 双工语音
- 富文本聊天体验
- 重型前端框架

一句话：

> 首版先解决“能听进去、能显示出来、能连续聊下去”，别先把自己搞复杂。

---

## 3. 产品判断

### 为什么单端优先

当前项目默认采用 **Tesla 车机单端方案**，原因如下：

- 用户核心诉求是 **在车里直接多轮对话**
- 手机端会显著增加复杂度（配对、同步、状态一致性、播放权属等）
- 当前没有证据证明 Tesla 浏览器路线必然走不通
- 只有当 Tesla 浏览器存在硬门槛时，才值得引入 fallback 架构

### 当前主输出为什么选择大字文本

与“车机端自动播报”相比，**大字文本显示** 的优先级更高，因为：

- 技术风险更低
- 更容易在老 Atom 车机上稳定落地
- 不受自动播放策略强约束
- 更利于先验证多轮对话主链路

TTS 在当前阶段是 **增强项**，不是成败前提。

## 4. 总体架构

```text
Tesla Browser
  -> 轻量 Web 前端
  -> App Server / API Gateway
  -> LLM / OpenClaw Gateway
  -> (Optional) TTS Provider
```

### 浏览器端职责

- 页面展示
- 文本输入
- 展示最近多轮消息
- 展示状态（待命 / 思考中 / 错误）

### 服务端职责

- 会话管理
- 多轮上下文组装
- 调用 OpenClaw / LLM
- 返回文本回复
- 记录日志与错误
- 可选 TTS 生成

## 7. 当前推荐开发顺序

### Step 1：先验证主链路
先做最小可运行原型，只验证：

- 页面打开
- 会话创建
- 文本输入
- LLM
- 大字文本显示
- 系统语音输入法辅助输入

### Step 2：再补稳定性
- 错误处理
- requestId 幂等
- 弱网恢复
- 最近消息恢复
- 页面性能优化

### Step 3：最后再考虑增强项
- TTS
- SSE
- 更好的状态流
- 更细的 UI 优化

## 8. 什么时候讨论 fallback

只有在出现以下硬门槛时，才考虑引入手机端或其它替代架构：

1. Tesla 浏览器无法稳定完成系统语音输入法辅助输入或文本发送
2. 输入或发送链路在正常网络下仍频繁失败
3. 页面无法承载基本多轮文本显示
4. 真实使用中存在无法通过工程手段缓解的浏览器限制

以下情况 **不构成 fallback 触发理由**：

- TTS 暂时没有
- 页面还不够精致
- 还没做性能优化就觉得不满意
- 开发过程中嫌麻烦

## 9. 对 Codex 的开发约束

如果将项目交给 Codex 或类似 AI 编程工具，必须遵守以下约束：

1. 不得引入重型前端框架
2. 不得擅自将项目改成多端协同方案
3. 不得让 TTS 反向绑架主链路
4. 不得把前端做成富前端 SPA 产品
5. 不得跳过测试、只追求“看起来能跑”

### 推荐工作方式
- 每次只做 1~3 个相关任务
- 每次改动前先读取文档
- 每次完成后输出变更摘要
- 优先保持架构一致性，而不是局部聪明

---

## 10. 当前项目一句话总结

> 这是一个为 Tesla 老款车机浏览器设计的极简 OpenClaw 客户端，MVP 重点验证“系统语音输入法辅助输入 + 大字文本多轮对话”是否能在车机单端稳定成立。

---

## 11. 下一步建议

建议按如下顺序推进：

1. 按验证计划做最小原型
2. 用真实 Tesla 浏览器做可行性测试
3. 只有在输入或显示主链路过不去时，再讨论 fallback
4. 如果主链路可行，再进入完整 MVP 开发

---

## 12. 一句话执行原则

> 先把主链路跑通，别提前把架构搞复杂。
